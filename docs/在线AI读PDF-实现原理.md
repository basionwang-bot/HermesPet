# 在线 AI 读 PDF —— 实现原理

> **版本**：v1.2.15 新功能（2026-05-24）
> **一句话**：让走网络的「在线 AI」也能读懂你拖进聊天框的 PDF——普通 PDF 秒读，拍照/扫描的也能自动认出文字。

---

## 一、为什么需要这个功能（问题出在哪）

HermesPet 里有四个「对话对象」，读本地文件的能力天生不一样：

| 对话对象 | 怎么读文件 | 能读 PDF 吗 |
|---|---|---|
| **Claude Code** | 本机子进程，有自己的 Read 工具 | ✅ 能（自己打开） |
| **Codex** | 本机子进程，有自己的 shell 工具 | ✅ 能（自己打开） |
| **Hermes** | 自部署，带本机文件工具 | ✅ 能（自己打开） |
| **在线 AI** | 走网络，底层是 DeepSeek / Kimi / 智谱等 OpenAI 兼容模型 | ❌ **读不了** |

**关键差别**：前三个都跑在你电脑上、有"手"去打开文件；而「在线 AI」只是把消息发到云端模型，模型在远端，**够不到你电脑里的文件**。

之前的逻辑是：拖 PDF 进在线 AI → 当成「文件附件」（专业叫 `file part`）发给云端模型 → 但这些文字模型**根本不认 PDF 这种二进制格式** → 要么报错 `file part media type not supported`，要么干脆装看不见、瞎答。

这就是你说的"在线 AI 读不了 PDF"。

---

## 二、核心思路（怎么解决）

> **不指望模型去读 PDF，而是在你电脑本地先把 PDF 拆成纯文字，再把文字直接塞进发给 AI 的消息里。**

文字模型读不了 PDF，但**文字它当然能读**。所以我们在本地做"翻译"：PDF → 纯文字 → 拼进消息。

PDF 分两种，走两条路：

```
                  拖一个 PDF 进「在线 AI」
                          │
                          ▼
              这个 PDF 有"文字层"吗？
            （电子书/导出的报告 = 有；
              拍照/扫描的合同   = 没有）
                  │                    │
              有 │                    │ 没有
                  ▼                    ▼
        ┌──────────────────┐  ┌──────────────────────┐
        │ 直接抽文字（秒出）│  │ 每一页当成图片，     │
        │  PDFKit 逐页读取  │  │ 用 OCR 认出里面的字   │
        └──────────────────┘  │ （复用项目现有 OCR）  │
                  │            └──────────────────────┘
                  │                    │
                  └─────────┬──────────┘
                            ▼
              把抽出来的文字拼进发给 AI 的消息
              （太长会自动截断，避免超出 AI 的容量）
                            ▼
                   云端模型读到的是纯文字，
                   于是能正常总结、问答了 ✅
```

## 三、具体怎么实现的

### 1. `PDFTextExtractor.swift` —— 专门抽文字的小工具（新建）

这是这次新加的文件，干一件事：**给它一个 PDF，还你一段纯文字**。内部逻辑：

1. **先试"文字层"**：用系统的 PDFKit 框架逐页读 `page.string`。电子书、从 Word/网页导出的 PDF 都自带文字层，这一步秒出。
2. **文字层是空的 → 判定为扫描版**：拍照、扫描出来的 PDF 本质是一张张图片，没有文字层。这时**逐页把页面渲染成图片**，再交给项目里已有的 `VisionOCR`（就是"圈选识别文字"用的那套）认出文字。
3. **防止内容太多撑爆 AI**：
   - 文字最多取 **6 万字**（约一本小册子）
   - 扫描版 OCR 最多处理 **30 页**（逐页识别比较慢）
   - 超出的部分截断，并在结尾标注"文档较长，已截断"
4. **完全抽不出**（加密 / 损坏的 PDF）→ 返回空，让 AI 转告你"这个 PDF 读不了"。

### 2. `OpenCodeHTTPClient.swift` 的 `buildParts` —— 把它接进发送流程

「在线 AI」发消息前，会把你的文字、图片、附件组装成一个数组发出去。这里加了一个**只针对 PDF 的分叉**：

```
组装要发送的内容时，逐个看附件：
  ├─ 如果是 PDF  → 调 PDFTextExtractor 抽成文字，作为「文本」塞进去
  └─ 其他文件    → 照旧当「文件附件」发（txt/csv 这类纯文本，opencode 本就能读）
```

也就是说：**只有 PDF 走"本地抽文字"这条特殊路，其他文件的逻辑一点没变**，互不影响。

## 四、几个关键的设计取舍（为什么这么选）

- **为什么在本地抽，而不是想办法让模型读？**
  因为云端模型够不到你的文件，且天生不支持 PDF 二进制。本地抽文字是最可靠、不依赖任何特定模型能力的办法——换哪家模型都能用。

- **为什么扫描版要做 OCR，而不是直接说"不支持"？**
  你拍照/扫描的合同、票据是很常见的需求。而且项目里**早就有一套 OCR**（"圈选识别文字"功能用的 `VisionOCR`），直接复用，成本很低，体验却完整很多。

- **为什么要设字数/页数上限？**
  AI 一次能处理的内容有上限（叫 context）。几百页的 PDF 全塞进去会超出上限、报错或很贵。所以截断到一个合理范围，并明确告诉你"截断了"。

- **其他文件（txt/csv）为什么不动？**
  那些是纯文本，云端的 opencode 引擎本来就能读，没必要多此一举。问题只出在 PDF 这种二进制格式上。

## 六、目前的局限（诚实说明）

- **只覆盖 PDF**：Word（docx）、Excel（xlsx）等其他二进制文档暂时还读不了——它们和 PDF 是同一个病（模型读不了二进制），但 PDFKit 抽不出它们的文字，需要另外的库。以后可以按同样的思路扩展。
- **超大 PDF 会截断**：几百页的文档只取前面一部分。
- **扫描版可能有少量错字**：OCR 识别不是 100% 准确，结果里会标注"可能有错字"提醒你。
